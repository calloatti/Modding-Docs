# Timberborn.AreaSelectionSystem

## Overview
The `Timberborn.AreaSelectionSystem` module is responsible for handling player input regarding spatial selections in the game world. This includes dragging the mouse to select multiple buildings, drawing roads, placing a line of dams, zoning forestry areas, or sculpting terrain. 

It converts 2D screen input (mouse clicks/drags) into 3D grid coordinates, determines what objects or terrain blocks lie within that selection, and passes that data back to the relevant building/tool systems.

---

## Key Components

### 1. `AreaSelectionController`
This is the core input processing loop for all area tools.
* **Mechanics:** It listens to the `InputService` for mouse events (`MainMouseButtonDown`, `MainMouseButtonHeld`, `MainMouseButtonUp`) and camera rotation events (`RotateButtonHeld`).
* **Rays:** It translates the mouse position into a Unity `Ray` via `_cameraService.ScreenPointToRayInGridSpace`. It tracks a `_startRay` (where the user first clicked) and an `_endRay` (where the user is currently dragging).
* **Callbacks:** It calls `previewCallback` while the mouse is being dragged, `actionCallback` when the mouse button is released, and `showNoneCallback` when the mouse is over UI.

### 2. `AreaBlockObjectPicker` & `AreaBlockObjectAndTerrainPicker`
These classes take the rays generated by the `AreaSelectionController` and convert them into actual game objects.
* **`AreaBlockObjectPicker`:** Uses `BlockObjectPicker` to find all `BlockObject` entities of type `<T>` within the drawn rectangle. It can be configured to pick upwards (e.g., selecting stacked buildings) or downwards.
* **`AreaBlockObjectAndTerrainPicker`:** Similar to the above, but also uses `ITerrainPhysicsService` to grab the underlying terrain coordinates `Vector3Int`. This is crucial for tools like the Demolish tool that can destroy both buildings and roads/terrain blocks simultaneously.

### 3. `AreaPicker`
This class handles *placement* logic rather than *selection* logic. It determines where ghosts/previews should appear when drawing out a line of buildings.
* **Layouts:** It reads the `PlaceableBlockObjectSpec.Layout` enum to determine how to draw the area. Supported layouts include:
  * `Single`: Just the block under the cursor.
  * `Rectangle`: A filled grid of blocks (e.g., planting carrots).
  * `Line`: A 1-block wide line (e.g., building paths).
  * `TwoSegmentLine`: An L-shape with one elbow joint (e.g., drawing power shafts).
* **Outputs:** It yields an `IEnumerable<Placement>` which contains the `Coordinates`, `Orientation`, and `FlipMode` for every preview block that should be spawned.

### 4. `SculptingTerrainPicker`
A specialized picker used exclusively by the Map Editor (and potentially late-game terraforming features) to handle the complex raycasting required for adding or removing 3D terrain blocks.

### 5. `RectangleBoundsDrawer`
Handles the visual representation of the selection box. It uses `MeshDrawer` to draw the colored borders and bottom tile highlights when you drag a rectangle across the terrain.

## How and When to Use This in a Mod

If you are creating a custom tool (e.g., a "Mass Upgrade" tool, a "Select all Beavers in Area" tool, or a custom building placement logic), you should inject and use the pickers from this DLL.

### Example: Creating a Custom Area Selection Tool
If you want to allow players to drag a box to select specific components, use `AreaBlockObjectPicker`.

```csharp
using System.Collections.Generic;
using Timberborn.AreaSelectionSystem;
using Timberborn.BlockSystem;
using UnityEngine;

public class MyCustomModTool
{
    private readonly AreaBlockObjectPicker _areaBlockObjectPicker;

    // Inject the factory to create a picker
    public MyCustomModTool(AreaBlockObjectPickerFactory pickerFactory)
    {
        // We want to pick objects from the ground up
        _areaBlockObjectPicker = pickerFactory.CreatePickingUpwards();
    }

    public void ProcessInput()
    {
        // Start processing. It will automatically handle dragging, 
        // raycasting, and camera movement during the drag.
        _areaBlockObjectPicker.PickBlockObjects<MyCustomComponent>(
            previewCallback: OnPreview,
            actionCallback: OnAction,
            showNoneCallback: OnShowNone
        );
    }

    // Called every frame while the player is dragging the mouse
    private void OnPreview(IEnumerable<BlockObject> blockObjects, Vector3Int start, Vector3Int end, bool selectionStarted, bool selectingArea)
    {
        // Highlight the hovered/selected objects
    }

    // Called the exact frame the player releases the mouse button
    private void OnAction(IEnumerable<BlockObject> blockObjects, Vector3Int start, Vector3Int end, bool selectionStarted, bool selectingArea)
    {
        foreach (BlockObject obj in blockObjects)
        {
            var myComponent = obj.GetComponentFast<MyCustomComponent>();
            // Perform your mod's action!
        }
    }

    private void OnShowNone()
    {
        // Clear preview highlights when mouse is over UI
    }
}
```

## Modding Insights & Limitations

* **Size Limits:** The system enforces limits on how large an area can be selected to prevent performance spikes or memory crashes. `AreaPicker` clamps the `IEnumerable` using `_maxBlocks`, which is loaded from the JSON file `AreaPickersSpec`.
* **Z-Axis Complexity:** Timberborn is a 3D game built on a grid. Note how `AreaSelector.GetSelectionEnd` carefully handles `VerticalOffset` and uses `_terrainPicker.FindCoordinatesOnLevelInMap`. If you write custom selection tools without using this system, you will likely struggle with players selecting objects on different vertical levels. Always rely on `AreaPicker` or `AreaBlockObjectPicker` to handle the Z-axis math for you.
