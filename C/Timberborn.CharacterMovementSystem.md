# Timberborn.CharacterMovementSystem

## Overview
The `Timberborn.CharacterMovementSystem` translates abstract pathfinding instructions (from `Timberborn.Navigation`) into smooth, continuous 3D movement and animations for characters. It handles the interpolation between grid coordinates, calculates movement speeds (including slowdowns near destinations), and manages the rotation and animation states (walking vs. running) of the character's 3D model.

---

## Key Components

### 1. `PathFollower` & `AnimatedPathFollower`
These two classes work together to turn a list of discrete path nodes into a smooth animation timeline.
* **`PathFollower`**: This acts as the logic controller. When `MoveAlongPath` is called (usually every AI tick), it calculates exactly how far the character should move based on `tickDeltaTime` and their current speed. 
    * It builds a list of `AnimatedPathCorner` structs, which are essentially keyframes containing a position, a timestamp (`Time`), and the speed at that moment. 
    * **Deceleration**: It includes a clever `GetSpeedLimitIfCloseToTarget` check. If the character is within 3 nodes of their final destination, it slightly adjusts their speed so they arrive exactly on the tick boundary, preventing them from overshooting or jittering at the end of their path.
* **`AnimatedPathFollower`**: This acts as the playback engine. It is completely decoupled from the AI tick rate. During the Unity `Update` loop, it receives the current `Time.time` and interpolates (`PlaceBetweenCorners`) the character's exact `CurrentPosition`, `CurrentSpeed`, and `CurrentXRotation` based on the timeline generated by the `PathFollower`.

### 2. `MovementAnimator`
This component acts as the central hub for movement-related events and state.
* **Update Loop**: Every frame (`Update(Time.deltaTime)`), it tells the `AnimatedPathFollower` to step forward in time. It then reads the resulting interpolated position/rotation and applies it to the `CharacterModel`.
* **Animator Sync**: It constantly checks the `CurrentSpeed` from the `AnimatedPathFollower`. If it changes, it updates the `"WalkingSpeed"` float parameter in the `CharacterAnimator`, ensuring the physical animation speed perfectly matches the visual displacement speed.
* **Persistence**: It implements `IPersistentEntity`. If the game saves while a beaver is mid-stride between two tiles, it saves the exact `Position` and the `LeftTimeInSeconds`. Upon loading, `AnimateMovementFromLoadedPositionToObjectPosition()` generates a temporary micro-path to smoothly walk the beaver from their loaded floating position back onto the exact center of the nearest grid tile.

### 3. `CharacterRotator`
This component calculates how the character should turn to face their movement direction.
* **Y-Axis (Heading)**: It calculates `CalculateYRotationSpeed` based on the angle to the target and the current movement speed. It features a "Corner Snap" optimization: if the character is very close to a corner (`<= 0.9f`) and the turn isn't too sharp (`<= 80f` degrees), it forces the rotation to perfectly align with the new path segment immediately, rather than sliding smoothly.
* **X-Axis (Pitch)**: If the character is running down stairs or ramps, it reads the `CurrentXRotation` from the `AnimatedPathFollower` and pitches the character's body forward or backward.

### 4. Speed Modifiers & `RunningProhibitor`
* **`IMovementSpeedAffector`**: An interface used to check if the character is slowed down (e.g., carrying heavy goods).
* **`RunningProhibitor`**: A state flag that explicitly forbids the "Running" animation state. `CharacterRotator.IsRunning()` checks both the `IMovementSpeedAffector` and this prohibitor. If a character cannot run, their X-axis pitch rotation is also forced to `0f` (they will stay upright on stairs instead of leaning into the sprint).

## How to Use This in a Mod

### Creating a Custom Speed Affector
If your mod introduces a new status effect (like "Injured" or "Walking through Mud") that should slow down a character and force them into a walking animation, you can implement `IMovementSpeedAffector`.

```csharp
using Timberborn.BaseComponentSystem;
using Timberborn.CharacterMovementSystem;

public class MudDebuff : BaseComponent, IMovementSpeedAffector, IUpdatableComponent
{
    private bool _inMud;

    // This interface property is automatically checked by the CharacterRotator
    // to determine if the character is allowed to use the running animation.
    public bool IsMovementSlowed => _inMud;

    public void Update()
    {
        // ... logic to determine if standing in mud ...
        _inMud = CheckIfInMud();
    }
}
```

Because `CharacterRotator` searches for this interface dynamically (`_movementSpeedAffector = GetComponent<IMovementSpeedAffector>()`), you just need to add your custom component to the character's JSON file; the movement system will pick it up automatically.

## Modding Insights & Limitations

* **Hardcoded Rotation Values**: `CharacterRotator` uses `private static readonly float` variables for its tuning (`XRotationSpeed = 330f`, `YRotationSpeed = 360f`, `CornerSnapMaxAngle = 80f`). Modders cannot customize how fast a specific character turns or how sharply they snap to corners via JSON specs. A giant slow-moving golem will snap to corners with the exact same math as a tiny beaver.
* **Component Fetching Limitation**: `CharacterRotator` uses `GetComponent<IMovementSpeedAffector>()` rather than `GetComponents`. `GetComponent` only returns the *first* instance of the interface it finds. Because the `GoodCarrier` component already implements `IMovementSpeedAffector`, any modded component implementing this interface on a beaver might be completely ignored if Unity happens to return `GoodCarrier` first.
* **Instant Movement Override**: In `PathFollower.MoveAlongPath`, if the calculated speed is `float.MaxValue` (used for teleportation or instant traversal like deep water tunnels), the system bypasses interpolation entirely and instantly places the character at the destination node.
